# Delete Nodes and Return Forest — Real-World Use Cases

Node deletion with child promotion — the core operation of this problem —
appears in React's reconciliation algorithm and compiler dead code elimination.

---

## 1. React Fiber Reconciler — Component Tree Node Deletion

React's virtual DOM is a tree of component nodes. When a component type
changes (e.g., `<div>` becomes `<span>`), React deletes the entire subtree
rooted at that node and creates a new one. The deleted node's children
either get deleted too or become roots of new component trees —
exactly the delete-nodes-and-return-forest operation.

### The structural parallel

```
Delete Nodes and Return Forest:
  deleteSet = {NodeA, NodeC}
  → NodeA's children become new roots
  → NodeC's children become new roots
  → remaining nodes stay connected

React reconciliation (when type changes):
  deletedComponents = {ComponentA, ComponentC}
  → ComponentA's children unmount and remount as new fiber roots
  → ComponentC's children get new parent fibers
  → surviving components stay in the fiber tree
```

React Fiber uses postorder traversal (completing children before parents)
during the "commit phase" — the same traversal order as this algorithm.

### React Fiber's postorder approach

React Fiber is a complete re-architecture of React's core reconciliation
algorithm. When an element's type has changed, React will dismiss the whole
subtree of that element and create a new tree recursively from scratch.
The reconciler does the work of computing which parts of a tree have changed;
the renderer then uses that information to actually update the rendered app.

Different component types are assumed to generate substantially different
trees — React will not attempt to diff them, but rather replace the old
tree completely.

## 2. Compiler — Dead Node Elimination (RVSDG IR)

Optimising compilers represent programs as graphs of computation nodes.
Dead code elimination (DCE) removes nodes whose results are never used —
exactly the node deletion problem. When a node is removed, its input
nodes may become new independent roots (no longer feeding into anything),
forming a forest of surviving computation subgraphs.

### The structural parallel

```
Delete Nodes and Return Forest:
  deleteSet = nodes to remove
  surviving children of deleted nodes → new forest roots
  deleted nodes return null to parents → severs edges

Dead Node Elimination in RVSDG:
  dead nodes = nodes whose outputs are unused
  surviving inputs of dead nodes → potentially become independent subgraphs
  removed in postorder (leaves before parents) → same traversal
```

### Algorithm in RVSDG (same logic as this problem)

Dead Node Elimination (DNE) removes all nodes that do not contribute
to the result of a computation. Dead nodes are generated by unreachable
and dead code from the input program. An operation is considered dead
when its results are either not used or only by other dead operations.
Thus, an output is dead if it has no users or all its users are dead.
A node is dead if all its outputs are dead. It follows that a node's
inputs are dead if the node itself is dead.

The algorithm proceeds bottom-up (postorder) — exactly like this problem:
process leaf nodes first, propagate deadness upward to parents.

## The Shared Algorithm

Both domains use the same postorder tree traversal:

```
postorder_delete(node, deleteSet, forest):
    if node is null: return null

    deleted = node.val in deleteSet
    if isRoot and not deleted: forest.add(node)

    node.left  = postorder_delete(node.left,  deleteSet, forest, isRoot=deleted)
    node.right = postorder_delete(node.right, deleteSet, forest, isRoot=deleted)

    return null if deleted else node
```

| Domain | Tree | Deleted nodes | Surviving children become |
|--------|------|--------------|--------------------------|
| This problem | Binary tree | Values in deleteSet | New roots in forest |
| React reconciler | Fiber component tree | Type-changed components | New fiber subtree roots |
| Compiler (RVSDG DCE) | IR computation graph | Dead (unused) nodes | Independent live subgraphs |
